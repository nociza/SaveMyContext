# Remote Access

SaveMyContext can expose a backend for remote Chrome extensions without asking the user to hand-configure every device. The host machine runs the backend and a managed tunnel. Remote browser profiles enroll with a connection string generated by the CLI.

The common case is simple:

1. run `smc install --remote` on the host machine
2. copy the emitted `smc_conn_1_...` string
3. paste it into the extension on each device you want to connect

## Goals

- one host machine can serve all of the user's devices
- the CLI manages exposure and enrollment
- the extension can be configured by pasting a single string
- stronger device-scoped options are available when the user wants them
- remote access still resolves to normal scoped SaveMyContext app tokens

## Exposure model

The transport layer and the authorization layer stay separate.

- **Transport:** a managed tunnel publishes the backend URL
- **Authorization:** the backend still issues and validates scoped SaveMyContext bearer tokens

The first managed provider is `tailscale-funnel`. The CLI configures Tailscale Funnel against the local backend and writes the resulting HTTPS URL into `public_url`.

## Connection strings

The CLI emits connection strings with this opaque format:

```text
smc_conn_1_<base64url-json>
```

The payload contains:

- bundle version
- backend URL
- grant id
- enrollment secret
- security level

The extension parses the string locally, calls the backend enrollment endpoint, receives a normal app token, and then stores only the backend URL plus the issued token.

## Security levels

SaveMyContext currently implements three connection-grant levels.

## Which mode to choose

### `smc share`

Use this when the devices are all yours and convenience matters most.

Examples:

- your desktop, laptop, and tablet
- your work Chrome profile plus your personal Chrome profile
- a host machine that you want to reuse across several devices quickly

### `smc invite --device <name>`

Use this when you want a separate, single-use string for one device.

Examples:

- one new laptop
- one Chrome profile you may revoke later
- one device you do not want sharing the same reusable string as everything else

### `smc invite --security per_device_code --device <name>`

Use this when you want the strongest current setup.

Examples:

- a work laptop
- a device you only partially trust
- a device where you want the copied string to be useless without a second approval step

### `shared`

One reusable connection string works on multiple devices.

- CLI prints a single connection string
- each successful redemption issues a new `ingest + read` app token
- easy setup
- broadest blast radius if the string leaks

### `per_device`

The CLI generates a single-use string for one target device.

- one string per device
- a redeemed string cannot be used again
- lower blast radius than `shared`

### `per_device_code`

The CLI generates a single-use string plus a separate verification code.

- one string per device
- one extra code must be entered during enrollment
- the connection string alone is not enough
- highest security level currently implemented

## Enrollment flow

### Shared bundle

1. User runs `smc share`
2. CLI ensures the backend is reachable through the managed tunnel and resolves or creates the local owner account
3. CLI prints one connection string
4. User pastes the string into extension settings on each device
5. Extension redeems the string for a normal app token

This is the best fit when one person is enrolling several of their own devices.

### Per-device bundle

1. User runs `smc invite --device laptop`
2. CLI prints a single-use string
3. User pastes that string into the target device's extension settings
4. Extension redeems it once and stores the issued token

This is the best fit when you want to treat each device as a separate enrollment event.

### Per-device bundle with code

1. User runs `smc invite --security per_device_code --device laptop`
2. CLI prints:
   - a single-use connection string
   - a separate verification code
3. User pastes the string and enters the verification code in the extension
4. Extension redeems both values and stores the issued token

This is the best fit when you want device-level approval instead of trusting the copied string by itself.

## Backend behavior

The backend stores connection grants in SQLite.

Each grant records:

- owner
- display name
- security level
- issued scopes
- secret hash
- optional second-factor hash
- single-use vs reusable redemption policy
- active / revoked status
- expiration time
- redemption count

Redeeming a grant creates a standard API token. The extension never uses the grant secret after enrollment.

## Public-surface rules

The capabilities endpoint is still unauthenticated, but it must not leak sensitive local filesystem paths to remote unauthenticated callers. Detailed storage paths remain visible only after authenticated requests or trusted local bootstrap access.

## CLI surface

The CLI manages both the tunnel and the grants.

### Simple commands

- `smc install --remote`
- `smc share`
- `smc unshare`
- `smc invite`
- `smc connections`
- `smc revoke`

Typical examples:

```bash
smc install --remote
smc share
smc invite --device ipad
smc invite --security per_device_code --device work-laptop
smc connections
smc revoke CONNECTION_ID
```

### Advanced commands

- `smc expose enable`
- `smc expose status`
- `smc expose disable`
- `smc expose bundle create`
- `smc expose bundle list`
- `smc expose bundle revoke`

`smc install --remote <level>` is the short convenience path that installs the service, enables remote exposure, and prints a first-use connection string in one step. `smc service install --start --expose <level>` remains available underneath it.

## Future extension points

The implemented model leaves room for stronger controls later:

- admin approval before token issuance
- TOTP-based second factor
- passkey or WebAuthn-backed device approval
- IP or provider identity hints from the tunnel layer
- revoking all tokens issued from a given grant

## Practical guarantee

The tunnel makes the backend reachable. The SaveMyContext token model still decides who can use it.

That means:

- tunnel misconfiguration does not grant API access by itself
- revoking a grant stops future enrollments
- revoking an app token cuts off an enrolled device
